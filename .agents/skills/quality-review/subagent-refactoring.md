# Subagent: Refactoring

> 检查方向：重复代码、过长函数/类、圈复杂度、可提取的公共逻辑、Magic Number。

## 检查清单

逐文件审查时，按以下清单逐项排查：

### 1. 重复代码
- 是否存在**跨文件的相似逻辑**（复制粘贴）可以提取为公共函数？
- 同一文件内是否存在相似的模式（只换了变量名或类型）？
- 多个 Inlet/Spout 子类中是否有重复的模板代码？

### 2. 函数/类长度
- 是否存在**超过 50 行的函数**？是否可以按职责拆分为更小的函数？
- 是否存在**超过 300 行的类**？是否可以按职责分离？
- `__init__` 方法是否过长（>20 行）？初始化逻辑是否需要提取？

### 3. 圈复杂度
- 是否存在**深层嵌套**（>3 层 if/for/while/try）？
- 长 if-elif 链是否可以用字典映射或策略模式替代？
- 是否存在可以通过 early return 减少嵌套的代码？

### 4. 可提取的公共逻辑
- 多个方法中的**前置/后置处理**是否可以提取为装饰器或上下文管理器？
- 错误处理模式是否重复（同样的 try/except 结构）？
- 配置/常量的初始化逻辑是否可以在模块级或类级复用？

### 5. Magic Number / Magic String
- 是否存在未命名的数字字面量（如 `60`、`0.8`、`1024`）？
- 模式字符串（`"serial"`, `"thread"`, `"eager"` 等）是否定义为常量或枚举？
- 默认值是否在多个位置重复定义？

### 6. 依赖注入
- 硬编码的依赖（如直接实例化具体类）是否可以通过参数注入？
- 是否存在可以通过构造函数注入来提升可测试性的地方？
