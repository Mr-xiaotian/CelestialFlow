# Subagent: Testability

> 检查方向：可测试性、接口隔离、Mock 友好度、依赖注入缺口。

## 检查清单

逐文件审查时，按以下清单逐项排查：

### 1. 接口隔离
- 类/模块的公开方法是否**过多**（>10 个），导致测试组合爆炸？
- 是否存在"做了太多事"的方法（查询 + 修改 + 持久化混在一起）？
- 内部状态是否可以通过公开方法充分观察和验证？

### 2. Mock 友好度
- 外部依赖（文件系统、网络、数据库、时间）是否通过参数注入而非硬编码？
- 是否存在 `time.time()`、`datetime.now()` 等难以在测试中控制的时间依赖？
- 是否存在 `os.cpu_count()`、`os.environ` 等难以在测试中隔离的系统依赖？
- 随机数/哈希等非确定性行为是否可以通过参数控制或 seed？

### 3. 依赖注入缺口
- 是否存在类内部 `new` 出来的具体实现（难以替换为 mock）？
- `__init__` 是否接受依赖作为参数，而非内部创建？
- 第三方库的客户端（如 Redis、HTTP）是否通过接口抽象？

### 4. 可观测性
- 关键内部状态（计数器、状态机、队列深度）是否有公开的查询方法？
- 是否存在只能通过 log/mock 才能验证的"不可见"副作用？
- 异步/线程执行的结果是否可以通过公开方法获取？

### 5. 测试基础设施
- 关键路径是否有对应的 fixture 或测试辅助函数？
- `conftest.py` 提供的 fixture 是否覆盖了常见测试场景？
- 是否存在需要大量样板代码才能设置测试的复杂对象？

## 区域特化提示

| 区域 | 重点关注 |
|------|---------|
| `stage/` | `TaskExecutor` 的 `ctree_client`/`observers` 注入、队列的可替换性、时间依赖 |
| `runtime/` | `TaskMetrics` 的状态查询、`TaskDispatch` 的线程池是否可替换 |
| `persistence/` | SQLite 路径的可配置性、Spout 线程的可测试性 |
| `web/` | FastAPI `TestClient` 兼容性、路由处理器的纯函数化程度 |
