# Subagent: Error & Edge

> 检查方向：异常处理覆盖、边界条件、资源泄漏风险、死代码。

## 检查清单

逐文件审查时，按以下清单逐项排查：

### 1. 异常处理覆盖
- 关键路径（I/O、网络、文件操作）是否有 try/except？
- except 块是否过宽（`except Exception` 吞掉了关键错误）？
- 是否存在**静默吞异常**（except 块只有 `pass` 或无意义的 log）？
- 抛出的异常类型是否准确表达了错误语义？

### 2. 边界条件
- 空容器（空 list/dict/queue）的处理是否安全？
- 数值计算的除零、溢出、负数等边界是否被考虑？
- 队列的 `put`/`get` 在队列满/空时行为是否正确？
- 字符串/路径的边界（空字符串、超长路径、特殊字符）？

### 3. 资源管理
- 文件句柄、数据库连接、线程等资源是否**保证释放**（`with` / `finally`）？
- 是否存在未关闭的资源导致泄漏？
- 守护线程（daemon thread）是否正确处理了退出信号？

### 4. 死代码
- 是否存在永远不会执行的分支（`if False`、不可达的 `return` 之后）？
- 是否存在定义了但从未调用的方法/函数？
- 是否存在被注释掉的大段代码（应该删除而非注释）？

### 5. 输入校验
- 公开 API 方法是否对输入参数做了必要的校验？
- 配置值（端口号、模式字符串等）是否在早期被校验而非延迟到运行时出错？

---

## 区域特化提示

| 区域 | 重点关注 |
|------|---------|
| `runtime/` | 队列操作异常、哈希计算边界、终止信号传播 |
| `stage/` | Executor 异常生命周期、重试逻辑的异常分支、任务失败后的状态一致性 |
| `persistence/` | 文件写入失败、SQLite 错误处理、spout 守护线程的异常退出 |
| `funnel/` | Spout `_spout()` 循环中的异常分支、TerminationSignal 处理 |
| `web/` | FastAPI 异常处理器、路由超时、WebSocket 断连 |
