# ログ永続化 (Log Persistence)

> 📅 最終更新日: 2026/05/09

`celestialflow.persistence` モジュールはマルチプロセス安全なログシステムを提供し、マルチプロセス環境でのログの統一収集、フォーマット、永続化の問題を解決することを目的としています。

コアコンポーネントは `LogSpout` と `LogInlet` です。

## アーキテクチャ設計

エラー永続化と同様に、ログシステムも **Logger-Listener** パターンを採用しています：

1.  **LogInlet（プロデューサー）**:
    -   各 Worker スレッドが保持するラッパークラス。
    -   豊富なセマンティックメソッド（`task_success`、`start_stage` など）を提供。
    -   ログメッセージとレベルをパッケージングし、スレッドセーフなキュー（`queue.Queue`）に投入。
    -   ログレベルベースのフィルタリングをサポートし、不要な通信を削減。

2.  **LogSpout（コンシューマー）**:
    -   独立したデーモンスレッドで実行。
    -   キューからログレコードを取得しファイルに書き込む。
    -   ログファイルのローテーションとフォーマットを統一管理。

## ログレベル

システムは以下の標準ログレベルをサポートしています（数値が大きいほど優先度が高い）：

-   **TRACE** (0): 最も詳細なトレース情報。キューの `put`/`get` 操作など。
-   **DEBUG** (10): デバッグ情報。プロセス終了、タスク入力など。
-   **SUCCESS** (20): 重要な操作の成功。タスク完了、分割成功など。
-   **INFO** (30): 一般情報。ステージの開始/終了、グラフ構造の出力など。
-   **WARNING** (40): 警告情報。タスクリトライ、キュー操作の例外など。
-   **ERROR** (50): エラー情報。タスク失敗、ループ例外など。
-   **CRITICAL** (60): 重大なエラー。

## LogSpout

`LogSpout` はログファイルの設定と書き込みスレッドを管理します。

### 初期化

```python
listener = LogSpout()
listener.start()
```

起動後、ログは `logs/task_logger({date}).log` ファイルに書き込まれます。

### ファイルパス

```text
logs/
└── task_logger(2023-10-27).log
```

## LogInlet

`LogInlet` は各コンポーネント向けの専用ログメソッドを提供し、ログ内容の構造化と一貫性を確保します。

### 初期化

```python
sinker = LogInlet(log_queue, log_level="SUCCESS")
```

-   `log_queue`: `LogSpout.get_queue()` が返すキュー。
-   `log_level`: この Inlet の最低ログレベルを設定。このレベル未満のログはキューに送信されない。

### 主要メソッド

#### タスク実行 (Executor)
-   `start_executor(...)`: エグゼキューターの起動を記録。
-   `end_executor(...)`: エグゼキューターの終了を記録（所要時間と統計情報を含む）。

#### ステージとレイヤー (Stage & Layer)
-   `start_stage` / `end_stage`: ステージのライフサイクルを記録。
-   `start_layer` / `end_layer`: staged スケジューリングにおける各レイヤーの実行を記録。

#### タスクライフサイクル (Task)
-   `task_input`: タスクが入力キューに入る。
-   `task_success`: タスクが正常に完了。結果の要約と所要時間を記録。
-   `task_retry`: タスクが失敗したがリトライをトリガー。
-   `task_error`: タスクが失敗しリトライ不可（最終失敗）。
-   `task_duplicate`: 重複タスクを検出。

#### タスクフロー (Flow)
-   `split_trace` / `split_success`: タスク分割プロセスを記録。
-   `route_success`: タスクルーティング結果を記録。
-   `put_item` / `get_item`: キューの低レベル操作を記録（TRACE レベル）。

#### レポーター (Reporter)
-   `push_*_failed`: Web サーバーへのデータプッシュ失敗の警告を記録。
-   `pull_*_failed`: Web サーバーからの設定取得失敗の警告を記録。

汎用的な `info()` や `debug()` ではなくこれらの専用メソッドを使用することで、生成されるログが読みやすく、機械的にも解析しやすくなります。
