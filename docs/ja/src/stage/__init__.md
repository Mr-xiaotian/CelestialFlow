# Stage モジュール

> 📅 最終更新日: 2026/04/24

Stage モジュールは CelestialFlow におけるタスク実行単位を定義します。基礎的なタスクエグゼキュータから複雑なタスクノードまでの完全な体系を提供し、タスクグラフを構築するための基本的なビルディングブロックです。

## モジュール概要

Stage モジュールは3つの階層のタスク実行単位を含みます：
1. **TaskExecutor**: 基礎タスクエグゼキュータ、単一タスクの実行ロジックを処理
2. **TaskStage**: 拡張型タスクノード、グラフ接続機能を追加
3. **定義済みノード**: 分流器、ルーターなど、一般的なタスクパターンの実装

これらのコンポーネントがタスク実行のコアを構成し、各コンポーネントは独立して使用することも、複雑なタスクフローに組み合わせることもできます。

## ファイル説明

### コアファイル

1. **core_executor.py** (`TaskExecutor`)
   - **役割**: 基礎タスクエグゼキュータ、単一タスクの実行、並行性制御、エラー処理、リトライ機構を処理
   - **主要機能**:
     - 同期/非同期タスク実行
     - エラー処理とリトライ戦略
     - タイムアウト制御
     - 進捗報告とログ記録
     - リソース管理とクリーンアップ

2. **core_stage.py** (`TaskStage`)
   - **役割**: 拡張型タスクノード、`TaskExecutor` を継承し、グラフ構造の接続機能を追加
   - **主要機能**:
     - 入力/出力キュー接続のサポート
     - 自動依存関係管理
     - グラフ構造を意識した実行
     - 並列・直列接続のサポート
     - 状態管理とライフサイクル制御

3. **core_stages.py** (定義済みノード: `TaskSplitter`, `TaskRouter`, `TaskRedisTransport`, `TaskRedisSource`, `TaskRedisAck`)
   - **役割**: 一般的なタスクパターンと Redis 統合ノードの事前実装、複雑なワークフローの構築を簡素化
   - **含まれるノード**:
     - `TaskSplitter`: 入力を複数のサブタスクに分配
     - `TaskRouter`: 条件に基づいてタスクを異なる下流ノードにルーティング
     - `TaskRedisTransport`: タスクを Redis キューに送信、クロス言語または分散実行に使用
     - `TaskRedisSource`: Redis キューからタスクを消費、タスクグラフの入力ソースとして使用
     - `TaskRedisAck`: Redis Worker の実行結果を受信し、タスク完了を確認

## モジュール関連

### 内部関連
- `TaskStage` は `TaskExecutor` を継承し、グラフ接続機能を拡張
- 定義済みノードはすべて `TaskStage` の特化実装
- すべてのノードは `TaskGraph` 内で組み合わせて使用可能

### 外部関連
- **Graph モジュールとの関連**: `TaskStage` は `TaskGraph` の基本的なビルディングブロック
- **Runtime モジュールとの関連**: `TaskQueue` を使用してノード間通信を行い、`TaskDispatch` に依存して実行
- **Utils モジュールとの関連**: ユーティリティ関数を使用してデータ処理と変換を実行
- **Persistence モジュールとの関連**: タスク状態の永続化保存をサポート

## 使用パターン

### 基本的な使用方法
1. **エグゼキュータの作成**: `TaskExecutor` を継承してカスタムビジネスロジックを実装
2. **ノードへのラップ**: `TaskStage` でエグゼキュータをラップし、グラフ接続をサポート
3. **グラフ構造の構築**: ノードを `TaskGraph` に追加し、依存関係を確立

### 高度な使用方法
1. **定義済みノードの使用**: `TaskSplitter`、`TaskRouter` などを直接使用して開発を簡素化
2. **ノードの組み合わせ**: 複数のノードを組み合わせて複雑なデータ処理パイプラインを構築
3. **カスタムノード**: `TaskStage` を継承してドメイン固有のノードタイプを作成

## 設計原則

### 単一責任
- 各 `TaskExecutor` は単一タイプのタスクのみを処理
- `TaskStage` はグラフ接続と状態管理に専念
- 定義済みノードは特定のデータ処理パターンを実装

### 組み合わせ可能性
- すべてのノードは統一されたインターフェースを使用し、自由に組み合わせ可能
- チェーン呼び出しと並列処理をサポート
- 入力/出力の型互換性チェック

### 拡張性
- 継承により簡単にカスタムノードを作成可能
- プラグイン式アーキテクチャをサポート
- 設定駆動で、コードを変更せずに動作を調整可能

## ベストプラクティス

1. **シンプルなタスク**: `TaskExecutor` を直接使用するか、継承して実装
2. **グラフノード**: 常に `TaskStage` またはサブクラスをグラフノードとして使用
3. **データ処理**: 定義済みノードを優先使用し、重複コードを削減
4. **エラー処理**: `TaskExecutor` レベルで堅牢なエラー処理を実装
5. **リソース管理**: `cleanup()` メソッドを正しく実装してリソースを解放
