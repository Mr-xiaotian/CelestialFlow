# Stage モジュール

Stage モジュールは、CelestialFlow におけるタスク実行ユニットを定義します。基本的なタスクエグゼキューターから複雑なタスクノードまでの完全な階層を提供し、タスクグラフを構築するための基本的なビルディングブロックです。

## モジュール概要

Stage モジュールには、3つのレベルのタスク実行ユニットが含まれています：
1. **TaskExecutor**: 単一タスクの実行ロジックを処理する基本タスクエグゼキューター
2. **TaskStage**: グラフ接続機能を備えた拡張タスクノード
3. **定義済みノード**: スプリッター、ルーターなど、一般的なタスクパターンの実装

これらのコンポーネントはタスク実行のコアを構成し、それぞれ独立して使用することも、複雑なタスクフローに組み合わせることもできます。

## ファイル説明

### コアファイル

1. **core_executor.py** (`TaskExecutor`)
   - **役割**: 単一タスクの実行、並行制御、エラーハンドリング、リトライ機構を処理する基本タスクエグゼキューター
   - **主な機能**:
     - 同期/非同期タスク実行
     - エラーハンドリングとリトライ戦略
     - タイムアウト制御
     - 進捗レポートとログ記録
     - リソース管理とクリーンアップ

2. **core_stage.py** (`TaskStage`)
   - **役割**: `TaskExecutor` を継承し、グラフ構造の接続機能を追加した拡張タスクノード
   - **主な機能**:
     - 入出力キュー接続のサポート
     - 自動依存関係管理
     - グラフ構造を認識した実行
     - 並列および直列接続のサポート
     - 状態管理とライフサイクル制御

3. **core_stages.py** (定義済みノード: `TaskSplitter`, `TaskRouter`, `TaskRedisTransport`, `TaskRedisSource`, `TaskRedisAck`)
   - **役割**: 一般的なタスクパターンと Redis 統合ノードの事前実装により、複雑なワークフローの構築を簡素化します
   - **含まれるノード**:
     - `TaskSplitter`: 入力を複数のサブタスクに分配します
     - `TaskRouter`: 条件に基づいてタスクを異なる下流ノードにルーティングします
     - `TaskRedisTransport`: クロス言語または分散実行のためにタスクを Redis キューに送信します
     - `TaskRedisSource`: タスクグラフの入力ソースとして Redis キューからタスクを消費します
     - `TaskRedisAck`: Redis Worker からの実行結果を受信し、タスクの完了を確認します

## モジュール関連

### 内部関連
- `TaskStage` は `TaskExecutor` を継承し、グラフ接続機能を拡張しています
- すべての定義済みノードは `TaskStage` の特殊化された実装です
- すべてのノードは `TaskGraph` 内で組み合わせて使用できます

### 外部関連
- **Graph モジュール**: `TaskStage` は `TaskGraph` の基本構成要素です
- **Runtime モジュール**: ノード間通信に `TaskQueue` を使用し、実行に `TaskDispatch` を使用します
- **Utils モジュール**: データ処理と変換にユーティリティ関数を使用します
- **Persistence モジュール**: タスク状態の永続的な保存をサポートします

## 使用パターン

### 基本的な使用
1. **エグゼキューターの作成**: `TaskExecutor` を継承してカスタムビジネスロジックを実装します
2. **ノードとしてラップ**: `TaskStage` を使用してエグゼキューターをラップし、グラフ接続を有効にします
3. **グラフの構築**: ノードを `TaskGraph` に追加し、依存関係を確立します

### 高度な使用
1. **定義済みノードの使用**: `TaskSplitter`、`TaskRouter` などを直接使用して開発を簡素化します
2. **ノードの組み合わせ**: 複数のノードを組み合わせて複雑なデータ処理パイプラインを構築します
3. **カスタムノード**: `TaskStage` を継承してドメイン固有のノードタイプを作成します

## 設計原則

### 単一責任
- 各 `TaskExecutor` は単一タイプのタスクのみを処理します
- `TaskStage` はグラフ接続と状態管理に専念します
- 定義済みノードは特定のデータ処理パターンを実装します

### 組み合わせ可能性
- すべてのノードは統一されたインターフェースを使用し、自由に組み合わせることができます
- チェーン呼び出しと並列処理をサポートします
- 入出力型の互換性チェック

### 拡張性
- 継承により簡単にカスタムノードを作成できます
- プラグインベースのアーキテクチャをサポートします
- 設定駆動型で、コード変更なしに動作を調整できます

## ベストプラクティス

1. **シンプルなタスク**: `TaskExecutor` を直接使用するか継承して実装します
2. **グラフノード**: グラフノードとして常に `TaskStage` またはそのサブクラスを使用します
3. **データ処理**: 重複コードを減らすために定義済みノードを優先的に使用します
4. **エラーハンドリング**: `TaskExecutor` レベルで堅牢なエラーハンドリングを実装します
5. **リソース管理**: リソースを解放するために `cleanup()` メソッドを適切に実装します
