# test_utils.py テストユーティリティ説明

> 📅 最終更新日: 2026/04/22

## テスト目的

`test_utils.py` はテストスイートの共有ユーティリティライブラリであり、ユニットテストとデモテスト用の統一されたテスト関数、データジェネレーター、ヘルパークラスを提供します。すべてのテストタスク関数は**純粋関数または制御された副作用関数**であり、予測可能で再現性のあるテスト動作を保証します。

## テスト範囲

### 1. 汎用計算関数
フレームワークのコア計算ロジックを検証するための数学関数：

| 関数 | 説明 | 特徴 |
|------|------|------|
| `fibonacci(n)` | 再帰フィボナッチ | 入力 `n <= 0` で `ValueError` をスロー、例外リトライのテスト用 |
| `fibonacci_async(n)` | 非同期再帰フィボナッチ | `async` 実行モードの検証 |
| `no_op(n)` | 恒等関数 | 計算オーバーヘッドなし、ベンチマーク対照用 |
| `sum_int(*num)` | 整数合計 | 複数引数展開の検証（`unpack_task_args=True`） |
| `add_one(num)` | 1加算 | 最も基本的な線形変換 |
| `sqrt(num)` | 平方根 | 浮動小数点演算の検証 |
| `square(x)` | 二乗 | `sleep(1)` 遅延付き、時間消費タスクのシミュレーション |

### 2. 例外境界付き関数

| 関数 | トリガー条件 | 例外タイプ |
|------|-------------|-----------|
| `add_offset(x, offset=10)` | `x > 30` | `ValueError` |
| `add_5` / `add_10` / `add_15` など | 同上（オフセット値が異なる） | `ValueError` |

これらの関数は以下の検証に使用：
- 例外キャプチャと統計
- リトライメカニズム（`ValueError` に対するリトライ設定）
- 部分的な失敗シナリオでのデータフロー連続性

### 3. Sleep バリアント

| 関数 | 説明 |
|------|------|
| `sleep_1(n)` | 同期1秒遅延 |
| `sleep_1_async(n)` | 非同期1秒遅延 |
| `add_one_sleep(n)` | 1秒遅延後に1加算、多条件例外付き |

### 4. URL 処理関数（demo_stages 専用）

クローラーワークフローをシミュレーションする関数群：
- `generate_urls_sleep` / `log_urls_sleep` / `download_sleep` / `parse_sleep`
- 各関数は4-6秒のランダム遅延を含み、I/O 集約型タスクをシミュレーション
- `download_to_file` は `requests` を使用して実際の HTTP リクエストを実行

### 5. 特殊クラス

#### `RouterWrapper`
`TaskRouter` テスト用のルーティングラッパー：
- 偶数は `a_tag` にルーティング
- 奇数は `b_tag` にルーティング
- フレームワークのリフレクション要件を満たすために `__name__` 属性の設定が必要

## 起こりうる問題と注意事項

### 1. `sleep` 付き関数がテストを遅くする
`square`、`add_offset`、`sleep_1` などの関数にはハードコードされた `sleep(1)` または4-6秒のランダム遅延が含まれています。ユニットテストではこれらの関数の使用を避け、`add_one`、`double` などの遅延なし関数を使用すべきです。

**解決策**:
- ユニットテストファイル（例: `test_executor.py`）は内部で独立した高速関数を定義
- デモテスト（例: `demo_structure.py`）は実際のワークロードをシミュレーションするために遅延関数を使用

### 2. `fibonacci` の再帰深度
`fibonacci(32)` は約200万回の再帰呼び出しを含み、以下を引き起こす可能性があります：
- Python の再帰深度制限（デフォルト1000）
- テスト実行時間の過度な長さ

**推奨**: ユニットテストでは `n <= 10` の小さな値のみ使用。

### 3. `download_to_file` のネットワーク依存
この関数は実際の HTTP リクエストを行い、以下のリスクがあります：
- ターゲット URL の無効化によるテスト失敗
- ネットワーク変動によるタイムアウト
- ローカルファイルシステムの権限問題（Windows で `/tmp/` パスが存在しない可能性）

**推奨**: デモ環境でのみ使用。ユニットテストでは `responses` または `httpx.MockTransport` でモックすべきです。

### 4. `random` の非決定性
`generate_urls` は `random.randint(1, 4)` を使用してランダムな URL 数を生成し、以下の結果：
- 実行ごとにタスク数が異なる
- 正確な成功/失敗アサーションが困難

**推奨**: 正確なアサーションが必要な場合、事前にランダムシード `random.seed(42)` を設定。

## 実行方法

このファイル自体にはテストケースは含まれず、インポートされる共有モジュールとして使用：
```python
from tests.test_utils import add_one, fibonacci
```

## 関連ファイル

- `tests/test_executor.py`: `add_one`、`double` などの高速関数を使用
- `tests/demo_executor.py`: `fibonacci`、`sleep_1` などのデモ関数を使用
- `tests/demo_stages.py`: URL 処理関数と `RouterWrapper` を使用
