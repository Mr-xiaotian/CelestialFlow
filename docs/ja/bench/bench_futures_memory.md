# bench_futures_memory.py ベンチマーク説明

> 📅 最終更新日: 2026/05/08

## 目的

スレッドプールシナリオにおいて、futures リストの「非クリーンアップ」と「定期クリーンアップ」の 2 つの戦略のメモリ使用量の差を純粋に比較する。CelestialFlow フレームワークに依存せず、futures 自体のメモリオーバーヘッドを分離して測定する。

## 背景

`ThreadPoolExecutor` はタスク送信後に `Future` オブジェクトを返す。すべての future をリストに追加して最終回収を待つ場合、リストはタスク数に応じて無限に増大する。完了済みの future を定期的にフィルタリングすることで、リストサイズを `max_workers * 2` に制限できる。

## テスト内容

### `dispatch_no_cleanup`
- すべての future をリストに追加し、クリーンアップを行わない
- 最終的に一括で `future.result()` を呼び出す

### `dispatch_with_cleanup`
- 追加のたびにリストサイズをチェックし、`max_workers * 2` に達したら完了済みの future をフィルタリング
- 最終的に残りの future に対して `result()` を呼び出す

## 主要パラメータ

| パラメータ | 値 | 説明 |
|------------|-----|------|
| `task_count` | 10K / 100K / 500K | タスク数 |
| `max_workers` | 20 | スレッドプールサイズ |
| タスク関数 | `noop(x)` | 即時リターン、計算オーバーヘッドなし |

## 実行方法

```bash
python bench/bench_futures_memory.py
```

### タスク規模とワーカー数の変更

`bench/bench_futures_memory.py` の `main()` 関数内で調整：

```python
def main():

    max_workers = 20  # スレッドプールサイズを調整
```

### 特定規模のみをテスト

```python
def main():
```

修正後に実行：

```bash

```

## ベンチマーク結果（実測）

> 環境：Windows、Python 3.10、max_workers=20

| タスク数 | モード | 所要時間 | ピークメモリ | 完了後メモリ |
|----------|--------|----------|-------------|-------------|
| 10,000 | no_cleanup | 0.351s | 17.94 MB | 0.20 MB |
| 10,000 | with_cleanup | 1.589s | 0.45 MB | 0.10 MB |
| 100,000 | no_cleanup | 5.507s | 177.01 MB | 0.18 MB |
| 100,000 | with_cleanup | 15.807s | 0.48 MB | 0.09 MB |
| 500,000 | no_cleanup | 30.232s | 883.51 MB | 0.16 MB |
| 500,000 | with_cleanup | 79.774s | 0.46 MB | 0.09 MB |

**主要な結論**：
- 非クリーンアップ時のピークメモリはタスク数に比例して増加：10K → 18 MB、100K → 177 MB、500K → 884 MB
- 定期クリーンアップ後はピークメモリが ~0.5 MB で一定、タスク数に依存しない
- 500K タスクシナリオではメモリを約 **1800x** 節約
- クリーンアップ操作には追加の時間オーバーヘッド（リストフィルタリング）があるが、実際のフレームワークではタスク自体に計算コストがかかるため、このオーバーヘッドは無視できる
