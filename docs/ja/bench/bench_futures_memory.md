# bench_futures_memory.py ベンチマーク説明

> 📅 最終更新日: 2026/05/08

## 目的

スレッドプールシナリオにおいて、futures リストの「クリーンアップなし」と「定期クリーンアップ」の 2 つの戦略間のメモリ使用量を純粋に比較する。CelestialFlow フレームワークに依存せず、futures 自体のメモリオーバーヘッドを分離して測定する。

## 背景

`ThreadPoolExecutor` はタスク送信後に `Future` オブジェクトを返す。すべての future をリストに追加して最終回収まで待つと、リストはタスク数に応じて無限に増大する。完了した future を定期的にフィルタリングすることで、リストサイズを `max_workers * 2` に制限できる。

## テスト内容

### `dispatch_no_cleanup`
- すべての future をリストに追加、クリーンアップなし
- 最後に一括で `future.result()` を呼び出し

### `dispatch_with_cleanup`
- 追加のたびにリスト長を確認、`max_workers * 2` に達したら完了済み future をフィルタリング
- 残りの future に対して最後に `result()` を呼び出し

## 主要パラメータ

| パラメータ | 値 | 説明 |
|------------|-----|------|
| `task_count` | 10K / 100K / 500K | タスク数 |
| `max_workers` | 20 | スレッドプールサイズ |
| タスク関数 | `noop(x)` | そのまま返す、計算オーバーヘッドなし |

## 実行方法

```bash
python bench/bench_futures_memory.py
```

## ベンチマーク結果（実測）

> 環境：Windows、Python 3.10、max_workers=20

| タスク数 | モード | 所要時間 | ピークメモリ | 完了後メモリ |
|----------|--------|----------|-------------|-------------|
| 10,000 | no_cleanup | 0.351s | 17.94 MB | 0.20 MB |
| 10,000 | with_cleanup | 1.589s | 0.45 MB | 0.10 MB |
| 100,000 | no_cleanup | 5.507s | 177.01 MB | 0.18 MB |
| 100,000 | with_cleanup | 15.807s | 0.48 MB | 0.09 MB |
| 500,000 | no_cleanup | 30.232s | 883.51 MB | 0.16 MB |
| 500,000 | with_cleanup | 79.774s | 0.46 MB | 0.09 MB |

**主要な結論**：
- クリーンアップなしではピークメモリがタスク数に比例して増加：10K → 18 MB、100K → 177 MB、500K → 884 MB
- 定期クリーンアップによりピークメモリはタスク数に関係なく約 0.5 MB で一定
- 500K タスクのシナリオでメモリ削減は約 **1800 倍**
- クリーンアップ操作には追加の時間オーバーヘッド（リストフィルタリング）があるが、実際のフレームワーク使用ではタスク自体に計算オーバーヘッドがあるため、このコストは無視できる
