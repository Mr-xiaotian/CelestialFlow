# bench_hash_memory.py ベンチマーク説明

> 📅 最終更新日: 2026/05/13

## 目的

SHA1 ハッシュ値を `str`（40 文字 hex）、`bytes`（20 バイト生データ）、`int`（160 ビット整数）の 3 つの型で `set` に格納した際のメモリ使用量とクエリ性能を比較し、`processed_set` のデータ型選択に根拠を提供する。

## テスト戦略

| 型 | 構築方法 | 単一オブジェクトサイズ | 説明 |
|----|----------|----------------------|------|
| `str` | `randbytes(20).hex()` | 81 B | 元の方式、40 文字の hex 文字列 |
| `bytes` | `randbytes(20)` | 53 B | 現在の方式、`hashlib.sha1().digest()` の直接出力 |
| `int` | `int.from_bytes(randbytes(20), 'big')` | 48 B | 最もコンパクトだが固定長セマンティクスを失う |

## 測定次元

- **単一オブジェクトサイズ**：`sys.getsizeof()` が返す Python オブジェクトメモリ
- **set 合計メモリ増分**：`tracemalloc` で N 個のオブジェクトを構築して set に追加した際のスナップショット差分
- **エントリあたりの平均オーバーヘッド**：合計増分 / N（set バケット、ポインタ等のオーバーヘッド含む）
- **構築時間**：N 回の `set.add()` の合計時間
- **クエリ遅延**：ヒット / ミス各 0.3 秒の定常状態測定、1 回あたりのナノ秒数を報告

## ベンチマーク結果（実測）

> 環境：Windows 11、Python 3.14、N=100,000

| 型 | 単一オブジェクト(B) | 合計メモリ(MB) | エントリあたり(B) | 構築(ms) | Hit(ns) | Miss(ns) |
|----|-------------------|---------------|------------------|---------|---------|----------|
| `str` | 81 | 11.73 | 123.0 | 60.40 | 110.5 | 112.2 |
| `bytes` | 53 | 9.06 | 95.0 | 56.23 | 112.7 | 112.7 |
| `int` | 48 | 8.58 | 90.0 | 69.05 | 121.9 | 112.4 |

### メモリ節約（str 基準）

| 型 | 合計メモリ | エントリあたりのオーバーヘッド |
|----|----------|-------------------------------|
| `bytes` | 77.2% | 77.2% |
| `int` | 73.2% | 73.2% |

**主要な結論**：
- `str→bytes` で約 23% のメモリ節約。百万タスク規模では約 27 MB の節約
- `bytes→int` でさらに約 5% のみ節約。ただし `int` は固定長セマンティクスがなく、デバッグ時に直感的でない
- 3 つの型でクエリ性能に有意な差はない（~112 ns）。型の切り替えは重複排除効率に影響しない
- **推奨：`bytes` を使用**：`hashlib.sha1().digest()` が直接返すため変換オーバーヘッドがゼロ

