# test_graph.py Test Documentation

> 📅 Last Updated: 2026/05/15

## Test Objective

Validates the correctness of `TaskGraph` and its subclasses (`TaskChain`, `TaskCross`, `TaskGrid`, `TaskComplete`) across multiple graph structures, including:
- DAG data flow construction and execution
- Fan-out, fan-in, error propagation
- Graph execution in async mode and thread mode
- stage_mode x execution_mode matrix combinations
- Graph structure analysis (DAG detection, level computation)
- Source node automatic inference
- Cyclic graph behavior
- Runtime summary statistics

### Key Test Case Details

#### `test_graph_dag_two_nodes`
- **Objective**: Data flow correctness for the simplest DAG (A -> B).
- **Assertions**: `stage1` succeeds 3 times, `stage2` succeeds 3 times (results correctly propagated).

#### `test_graph_fan_out`
- **Objective**: One upstream node distributes to multiple downstream nodes.
- **Assertions**: source 2, sink_a 2, sink_b 2.

#### `test_graph_fan_in`
- **Objective**: Multiple upstream nodes converge to one downstream node.
- **Assertions**: merge node receives 4 tasks (2 + 2).

#### `test_graph_error_propagation`
- **Objective**: Error tasks do not block the overall flow and are not propagated downstream.
- **Design**: `stage1` processes `[1, 50, 2]`, where `50` triggers `ValueError`.
- **Assertions**: `stage1` succeeds 2, fails 1; `stage2` only receives 2 successful tasks.

#### `test_complete_structure`
- **Objective**: Basic startup and structure detection for a complete graph (non-DAG).
- **Note**: Complete graph auto-injection of termination signals triggers `RuntimeWarning`; nodes may shut down prematurely. The test only verifies `isDAG == False` and that each node processed at least its own input.

#### `test_layer_computation`
- **Objective**: Correct topological level computation for a DAG.
- **Design**: A -> B -> C three-layer chain.
- **Assertions**: A at level 0, B at level 1, C at level 2.

#### `test_graph_summary_counts`
- **Objective**: Accurate summary statistics generated by `collect_runtime_snapshot()`.
- **Note**: The test does not enable `TaskReporter`; `collect_runtime_snapshot()` must be called **manually** to generate the summary.

## Potential Issues and Notes

### 1. Timeout Risk in Thread Mode
The default `stage_mode` for `TaskCross`, `TaskGrid`, and `TaskComplete` is `"thread"`. For larger grids, thread startup and scheduling overhead may exceed the test timeout threshold.

**Current test scale**:
- `test_cross_structure`: 2x3 grid, 4 threads
- `test_grid_structure`: 2x2 grid, 4 threads
- `test_complete_structure`: 3 nodes, 3 threads

The above scales pass reliably within the 180-second timeout.

### 2. Termination Signal Behavior in Non-DAG Graphs
`TaskComplete`, `TaskLoop`, `TaskWheel`, and other cyclic structures trigger a framework warning when `put_termination_signal=True`:
```
RuntimeWarning: Early injection of termination signals in a non-DAG graph ...
```

This occurs because in eager mode, nodes may shut down immediately upon receiving a termination signal while subsequent tasks from other nodes are still in the queue. Current tests have adjusted assertions to `>= 1` instead of exact values to accommodate this behavior.

**Recommendation**: For precise task counting in cyclic graphs:
1. Use `put_termination_signal=False`
2. Inject termination signals via the Web interface or externally
3. Run in staged mode (cyclic graphs do not support staged mode)

### 3. Manual Invocation of `collect_runtime_snapshot`
Data returned by `get_graph_summary()` comes from the last `collect_runtime_snapshot()` snapshot. In production, `TaskReporter` calls this periodically and automatically; however, in unit tests (without reporter enabled), it must be called manually.

Omitting this call will cause `get_graph_summary()` to return an empty dictionary or stale data.

Covers all 6 combinations of `stage_mode` (serial/thread) x `execution_mode` (serial/thread/async):

| Test Case | stage_mode | execution_mode |
|-----------|-----------|----------------|
| `test_serial_serial` | serial | serial |
| `test_serial_thread` | serial | thread |
| `test_serial_async` | serial | async |
| `test_thread_serial` | thread | serial |
| `test_thread_thread` | thread | thread |
| `test_thread_async` | thread | async |

Each test case uses a two-node DAG with 5 input tasks and verifies that both stages succeed 5 times each.

#### `test_source_stages_linear`
- **Objective**: In a linear graph (A -> B -> C), the only source is the head node A.

- **Objective**: Two entry points converging into one; source nodes are the two entry nodes.

### `TestCyclicGraph` Detailed Test Case Descriptions

#### `test_cyclic_isDAG_false`
- **Objective**: A cyclic graph (s1 -> s2 -> s3 -> s1) has `isDAG` as `False`.
- **Note**: Uses `put_termination_signal=True` to terminate the cycle.

#### `test_cyclic_layers`
- **Objective**: Cycle nodes (s1, s2, s3) share the same level; tail node s4 is at cycle level + 1.

### 4. Task Function Requirements
All test task functions and parameters use top-level functions and basic types to ensure compatibility.

### 5. Thread Safety
In current tests, all thread creation is managed internally by `TaskGraph.start_graph()`, requiring no additional protection.

## How to Run

```bash
# Run all tests
pytest tests/test_graph.py -v

# Structure tests only (most time-consuming, involves multi-threading)
pytest tests/test_graph.py::TestTaskGraphStructure -v

# Analysis tests only (fastest, no task execution)
pytest tests/test_graph.py::TestTaskGraphAnalysis -v
```
