# test_graph.py Test Documentation

> 📅 Last updated: 2026/04/22

## Test Purpose

Validates `TaskGraph` and its subclasses (`TaskChain`, `TaskCross`, `TaskGrid`, `TaskComplete`) across multiple graph structures, including:
- DAG data flow construction and execution
- Fan-out, fan-in, error propagation
- Graph structure analysis (DAG detection, layer computation)
- Runtime summary statistics

## Test Scope

| Test Class | Cases | Coverage |
|------------|-------|----------|
| `TestTaskGraphBasic` | 4 | Two-node DAG, fan-out, fan-in, error propagation |
| `TestTaskGraphStructure` | 4 | Chain, Cross, Grid, Complete structures |
| `TestTaskGraphAnalysis` | 2 | DAG detection, layer computation |
| `TestTaskGraphSummary` | 1 | Summary statistics |

### Key Test Case Details

#### `test_graph_dag_two_nodes`
- **Purpose**: Data flow correctness of the simplest DAG (A -> B).
- **Assertions**: `stage1` succeeds 3, `stage2` succeeds 3 (results correctly propagated).

#### `test_graph_fan_out`
- **Purpose**: One upstream node distributes to multiple downstream nodes.
- **Assertions**: source 2, sink_a 2, sink_b 2.

#### `test_graph_fan_in`
- **Purpose**: Multiple upstream nodes converge into one downstream node.
- **Assertions**: merge node receives 4 tasks (2 + 2).

#### `test_graph_error_propagation`
- **Purpose**: Error tasks do not block the overall flow and are not propagated downstream.
- **Design**: `stage1` processes `[1, 50, 2]`, where `50` triggers `ValueError`.
- **Assertions**: `stage1` succeeds 2, fails 1; `stage2` receives only 2 successful tasks.

#### `test_complete_structure`
- **Purpose**: Basic startup and structure detection of a complete graph (non-DAG).
- **Note**: Complete graphs with auto-injected termination signals trigger `RuntimeWarning`; nodes may shut down prematurely. The test only verifies `isDAG == False` and that each node processed at least its own input.

#### `test_layer_computation`
- **Purpose**: DAG topological level computation is correct.
- **Design**: A -> B -> C three-layer chain.
- **Assertions**: A at level 0, B at level 1, C at level 2.

#### `test_graph_summary_counts`
- **Purpose**: Summary statistics generated by `collect_runtime_snapshot()` are accurate.
- **Note**: `TaskReporter` is not enabled in this test; `collect_runtime_snapshot()` must be **called manually** to generate the summary.

## Potential Issues and Notes

### 1. Timeout Risk in Multiprocessing Mode
The default `stage_mode` for `TaskCross`, `TaskGrid`, `TaskComplete` is `"process"`. On Windows, each `multiprocessing.Process` startup takes approximately 100-300ms. For larger grids (e.g., 4x4), total startup time may exceed the test timeout threshold.

**Current test scale**:
- `test_cross_structure`: 2x3 grid, 4 processes
- `test_grid_structure`: 2x2 grid, 4 processes
- `test_complete_structure`: 3 nodes, 3 processes

These scales pass reliably within the 180-second timeout.

### 2. Termination Signal Behavior in Non-DAG Graphs
`TaskComplete`, `TaskLoop`, `TaskWheel` and other cyclic structures trigger a framework warning when `put_termination_signal=True`:
```
RuntimeWarning: Early injection of termination signals in a non-DAG graph ...
```

This is because in eager mode, nodes may shut down immediately upon receiving a termination signal while tasks from other nodes are still in the queue. Current tests have adjusted assertions to `>= 1` rather than exact values to accommodate this behavior.

**Recommendation**: To precisely test task counts in cyclic graphs:
1. Use `put_termination_signal=False`
2. Inject termination signals via the Web interface or externally
3. Run in staged mode (cyclic graphs do not support staged mode)

### 3. Manual Invocation of `collect_runtime_snapshot`
Data returned by `get_graph_summary()` comes from the last `collect_runtime_snapshot()` snapshot. In production, `TaskReporter` calls it periodically and automatically; but in unit tests (without reporter enabled), it must be called manually.

Omitting this call causes `get_graph_summary()` to return an empty dictionary or stale data.

### 4. `stage_mode="process"` and Pickle
All tests using `stage_mode="process"` require their task functions and arguments to be picklable. Current tests use top-level functions and basic types, so this is not an issue.

### 5. Windows Process Spawn Overhead
Windows uses `spawn` by default to create processes (Linux/macOS use `fork`). `spawn` re-imports the main module; if the `if __name__ == "__main__":` guard is not properly in place, recursive process creation may occur.

**Current tests**: All process creation is managed internally by `TaskGraph.start_graph()`, requiring no additional guards.

## How to Run

```bash
# Run all tests
pytest tests/test_graph.py -v

# Structure tests only (most time-consuming, includes multiprocessing)
pytest tests/test_graph.py::TestTaskGraphStructure -v

# Analysis tests only (fastest, no task execution)
pytest tests/test_graph.py::TestTaskGraphAnalysis -v
```
