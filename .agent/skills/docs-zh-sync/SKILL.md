# Docs Zh Sync

用于把代码与 `docs/zh-CN` 文档持续对齐的技能。

当用户提出以下需求时，立即调用本技能：

- 同步、补全、刷新中文文档
- 依据代码更新 `docs/zh-CN`
- 检查 `src/`、`bench/`、`tests/`、`demo/` 后批量修正文档
- 发现文档过期、缺页、路径不一致，要求按代码现状修复

## 目标

检查以下目录中的源码与示例：

- `src/`
- `bench/`
- `tests/`
- `demo/`

覆盖以下文件类型：

- `*.py`
- `*.ts`
- `*.html`
- `*.css`

然后把对应内容更新到 `docs/zh-CN/` 下相同语义、相同相对路径的文档中，文档文件统一为 `*.md`。

## 路径映射规则

默认使用"代码路径 -> 中文文档路径"的镜像规则。

### 根目录映射

- `src/celestialflow/...` -> `docs/zh-CN/src/...`
- `bench/...` -> `docs/zh-CN/bench/...`
- `tests/...` -> `docs/zh-CN/tests/...`
- `demo/...` -> `docs/zh-CN/demo/...`

### 后缀映射

- `foo.py` -> `foo.md`
- `foo.ts` -> `foo.md`
- `foo.html` -> `foo.md`
- `foo.css` -> `foo.md`
- `__init__.py` -> `__init__.md`

### 示例

- `src/celestialflow/runtime/util_errors.py` -> `docs/zh-CN/src/runtime/util_errors.md`
- `src/celestialflow/web/static/ts/main.ts` -> `docs/zh-CN/src/web/static/ts/main.md`
- `src/celestialflow/web/templates/index.html` -> `docs/zh-CN/src/web/templates/index.md`
- `src/celestialflow/web/static/css/dashboard.css` -> `docs/zh-CN/src/web/static/css/dashboard.md`
- `tests/runtime/test_queue.py` -> `docs/zh-CN/tests/runtime/test_queue.md`
- `demo/demo_graph.py` -> `docs/zh-CN/demo/demo_graph.md`

如果项目里已经存在旧版但不镜像的中文文档路径，优先以"镜像路径"作为目标；必要时说明发现了旧路径遗留问题。

## 分区执行策略

优先按区域拆分任务，并尽量使用 `subagent` 并行处理。推荐最少拆成以下 5 个区域：

1. `src/` 核心 Python 模块
2. `src/` Web 前端资源（`ts/html/css`）
3. `bench/`
4. `tests/`
5. `demo/`

如果代码量较大，可继续细分 `src/`：

- `src/celestialflow/runtime|graph|stage|funnel`
- `src/celestialflow/observability|persistence|utils`
- `src/celestialflow/web/static/ts|css|templates`

每个 `subagent` 负责：

- 扫描所属区域的代码文件
- 找到对应的 `docs/zh-CN` 目标文档
- 对照现有文档判断"更新 / 新建 / 保留"
- 只根据代码事实写文档，不凭空补设定
- 返回该区域的变更清单、风险点、缺失项

如果当前环境不支持真正的 `subagent`，则按同样的区域边界顺序串行执行，但在思考和输出中保持分区。

### 执行效率建议

- **推荐三阶段流水线**：审计 → 写入 → 改进。先全面了解代码与文档差距，再批量写入，最后按维度优化。每次切换阶段前用 `diagnostics` 确认无新问题。不要在一个 agent 里同时审计和写入大量文件，容易遗漏。
- **并行规模**：一个 100+ 文件的文档项目，不用 sub-agent 几乎不可能在合理时间内完成。审计阶段用 3-5 个 agent 并行，写入阶段用 4 个 agent 并行。
- **每个 agent 上限 15-20 个文件**：文件过多时 agent 难以维持对每个文件的细致关注，超过此数量应进一步拆分区域。

## 推荐工作流

1. 先列出本次需要覆盖的代码文件，按区域分组。
2. 对每个区域读取源码与现有中文文档。
3. 判断文档是否存在以下问题。以下是经大量审计总结的最常见 doc-code mismatch 类型，按出现频率从高到低排列，每次审计时优先排查：

   | 频率 | 模式 | 典型表现 | 排查方法 |
   |:----:|------|---------|----------|
   | 🔴 极高 | **虚构功能** | 文档声称支持"日志轮转""多后端存储""内存监控"等，源码无对应实现 | 对照源码每个公开方法逐一确认 |
   | 🔴 极高 | **遗漏新增类/函数/异常** | 源码重构后新增了 9 个异常类或 5 个方法，文档完全未提及 | 用 `grep "class \|def "` 对照文档中的函数列表 |
   | 🟠 高 | **参数名过期** | `current_name` → `out_name`/`in_name`，`max_retries` → 实际不存在 | 关注源码 `__init__` 签名 vs 文档中的参数表 |
   | 🟠 高 | **函数归属错误** | `toggleDarkTheme()` 实际在 `main.ts` 中，文档错误记在 `utils.ts` 下 | 用 `grep` 确认函数定义位置 |
   | 🟡 中 | **API 端点虚构** | 文档列了 `/api/pull_history_limit`，源码中无此路由 | grep 源码中的路由注册或 URL 常量 |
   | 🟡 中 | **算法/性能解释错误** | 用"GIL 竞争"解释耗时差异，实际是算法复杂度 O(φⁿ) vs O(n) | 并行读取相关源文件，理解实际算法 |
   | 🟢 低 | **旧路径遗留** | 文档在 `tests/` 根目录下平铺，但源码已迁移到子目录 | 按镜像规则全量核对路径 |
   | 🟢 低 | **路径不镜像** | 文档路径不符合"代码路径 → 中文文档路径"的镜像规则 | 同上 |

   此外也不应忽略：接口签名过期、示例与当前代码不一致、仍在描述已删除能力。

4. 更新或创建对应的 `docs/zh-CN/**/*.md`。
   - 对于更新或者新创建的文档，确保 `最后更新日期` 为当前日期（YYYY/MM/DD）。
   - 对于没有修改的文档，保持 `最后更新日期` 不变。
5. 完成后给出区域级汇总：
   - 更新了哪些文档
   - 新建了哪些文档
   - 哪些旧文档路径可能需要后续迁移
   - 哪些代码仍缺少足够上下文，暂不应编造说明

## 文档写作规则

- 默认使用中文。
- 以代码为唯一事实来源，必要时结合同目录已有文档风格。
- 重点写"职责、结构、关键函数、关键类、数据流、异常、配置、注意事项"。

### 按文档类型差异化

- 对 `tests/` 文档，重点说明测试覆盖目标、关键场景、断言意图，不要把测试逐行复述。使用 `pytest -k "pattern" -v` 格式的选择性运行命令作为使用示例，不要写 `python` 代码块。
- 对 `bench/` 文档，重点说明 benchmark 比较对象、衡量指标、实验目的、结论边界。使用示例应展示如何修改 `N`、`max_workers`、`COUNT` 等参数来调整基准测试规模。
- 对 `demo/` 文档，重点说明演示目的、运行方式、预期行为、依赖。使用示例应展示终端输出的 mock 示例，帮助用户理解运行后应看到什么。
- 对 `ts/html/css` 文档，重点说明页面职责、模块交互、DOM/样式作用域、与后端接口或其他前端模块的关系。

### 三类文件的特殊写法

- **`__init__.py` → `__init__.md`**：作为模块入口文档，应列出模块全部公开导出符号，并按子模块分组说明。包根入口（`src/celestialflow/__init__.py`）还应有完整的 `__all__` 列表和完整的包导入使用示例。
- **`util_*.py` → `util_*.md`**：工具类文档应重点说明每个公开函数的用途、参数、返回值和使用示例。函数数量较少时可以逐函数列表，数量多时建议表格化。
- **`conftest.py` → `conftest.md`**：测试固件文档应说明提供的 fixture 及其用途，不需要运行方式，也不需要日期。

## 文档质量要求

- 可以适当引用代码片段以避免单一文字模式，但不要整段照抄。
- 不要虚构未实现的功能。
- 不要仅把函数名罗列出来而没有解释。
- 不要遗漏最近的重构结果，尤其是重命名、异常类型变化、字段变化、状态键变化。
- 如果发现文档内容与代码冲突，按代码修正文档，并在最终总结里指出冲突点。
- 不要轻易删除原有内容：原文档中已有的信息允许修改、补充、重组，但除非该信息已被代码证实为完全错误或彻底废弃，否则不要删除。宁可保留旧信息并标注"已变更"或"待确认"，也不要让有价值的历史说明丢失。

### 使用示例要求

- **完整可运行是底线**：每个 Python/TS 文档的示例必须包含必需的 `import` 语句，用户可以复制粘贴运行（或稍作调整即可运行）。
- **一个示例胜过十行描述**：优先用带语言标注的代码块展示用法，而不是用文字反复解释。用 `\`\`\`python` 或 `\`\`\`ts` 包裹。
- **覆盖典型路径**：至少展示该模块最常见的 2-3 种使用场景，让用户快速掌握核心用法。

### 质量评估框架

当用户追问文档质量时，可使用以下 8 维评分框架进行系统化评估，先逐模块打分再汇总为项目均分：

| 维度 | 核心/辅助 | 权重 | 评判标准 |
|------|:---------:|:----:|---------|
| 🎯 内容准确度 | ★ 核心 | 2x | 与代码事实一致，无虚构功能、无错误描述 |
| 📐 结构完整性 | ★ 核心 | 2x | 有清晰的骨架层次、标题递进 |
| 🎨 表达丰富度 | ★ 核心 | 2x | Mermaid 图、表格、引用块、代码块等的运用密度 |
| 🔍 接口覆盖度 | ★ 核心 | 2x | 关键类、函数、参数、返回值、异常的覆盖率 |
| 📝 使用示例 | ☆ 辅助 | 1x | 代码示例的可运行性、完整性、场景覆盖 |
| 👀 可读性 | ☆ 辅助 | 1x | 语言流畅度、信息密度、排版舒适度 |
| 🔄 维护状态 | ☆ 辅助 | 1x | 更新日期是否最新、重构结果是否同步 |
| ⭐ 整体 | - | - | 上述维度加权综合（满分 10） |

> 核心维度的 0.5 分变化比辅助维度的同等变化对综合分影响大一倍。

## 文档表达形式

文档不仅是信息的载体，更是开发者阅读体验的一部分。应善用丰富的表达形式，使阅读文档成为乐趣：

- **流程图 (`flowchart`)**：最通用的 Mermaid 类型，用于数据流、架构拓扑、处理步骤。默认使用 `LR`（左→右）或 `TD`（上→下）方向。用 `subgraph` 对节点分组（如生产者/消费者、前端/后端），大幅提升可读性。
- **时序图 (`sequenceDiagram`)**：适合 API 交互和模块间调用顺序，保留 `loop` 和 `alt` 控制结构展示重复和条件分支。
- **类图 (`classDiagram`)**：适合展示核心类关系和继承结构，当存在基类和多个子类时优先使用。
- **状态图 (`stateDiagram-v2`)**：适合展示生命周期和状态机，如 Spout、Executor 等有明确状态转换的组件。
- **节点命名技巧**：用简洁英文作为主名称（如 `LogInlet`、`queue.Queue`），用 `<br/>` 换行补充中文简注（如 `LogSpout<br/>日志消费者`）。
- **表格**：适合对比、配置项罗列、参数说明等结构化信息，比纯文字更清晰。
- **ASCII 图**：Mermaid 占主导，但简单的 ASCII 图可以保留作为快速参考和终端输出示例。
- **列表与层级**：合理使用有序/无序列表、嵌套层级，让信息层次分明。
- **代码块**：关键示例使用带语言标注的代码块（`\`\`\`python`、`\`\`\`ts`、`\`\`\`bash`），方便语法高亮。
- **引用块**：用 `>` 引用块突出重要提示、注意事项、设计决策背景。

原则：形式服务于内容。不要为了花哨而堆砌图表，但也不要让文档变成一堵文字墙。好的文档应该让读者愿意读、容易读、读完有收获。

## 建议文档骨架

可按文件类型灵活调整，但通常保持以下结构：

```md
# 文件或模块标题

> 最后更新日期: YYYY/MM/DD

## 作用

## 核心对象 / 核心函数

## 关键流程

## 重要细节

## 使用示例 / 测试重点 / 运行方式

## 注意事项
```

## 排除项

除非用户明确要求，否则通常不处理以下内容：

- `docs/en/`
- `docs/ja/`
- 生成产物，如 `src/celestialflow/web/static/js/*.js`
- 第三方依赖锁文件、图片、二进制资源

## 交付要求

完成后输出：

- 本次扫描的区域
- 更新或新建的文档路径
- 发现的旧文档路径或结构不一致问题
- 仍待人工确认的歧义点

如果只完成了部分区域，要明确列出已完成范围和剩余范围。
